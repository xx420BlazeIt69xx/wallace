# Next step ÔÇö T6040 cpufreq bring-up (Stage B, item 1)

Written 2026-07-10, immediately after Stage A completed. This is the detailed
plan for the roadmap's Stage B item 1. Companion: `docs/ROADMAP.md`, and the archived
`done/t6040-bringup-plan.md` (Stage A).

## Why this is next

The boot log already names exactly one known gap: `cpufreq: Chip 0x6040 is
unsupported` (emitted from `src/cpufreq.c` `cpufreq_get_clusters()` /
`cpufreq_get_features()` default cases). It's the roadmap's highest-leverage
*local* work: no reverse engineering needed, T6040 looks structurally identical
to T6031 (M3 Max), and DVFS/APSC init is a prerequisite for sane clocks under
every later coprocessor bring-up. Small, self-contained, upstreamable.

**Strong prior that T6040 == T6031 here:** the `t6031_clusters` table is exactly
`ECPU0 + PCPU0 + PCPU1` (1 E-cluster + 2 P-clusters), which is precisely the
topology we confirmed by MPIDR this session (E-cl Aff1=0, P-cl0 Aff1=1, P-cl1
Aff1=2; see memory `t6040-smp-topology`). cpufreq operates per-cluster, so the
differing per-cluster core counts (M4 Pro 4E+5P+5P) don't matter to this table.

**The one real risk:** cluster PSTATE register *base addresses*. The T6031 table
uses `0x210e00000 / 0x211e00000 / 0x212e00000`. But T6040's MMIO map is NOT
identical to M3's ÔÇö `EARLY_UART_BASE` is `0x429200000` on T6040 vs the M3 value,
so the arm-io/cluster physical base may differ. **These bases must be verified
against T6040's own ADT before we trust the copy.** This is the crux of the plan;
everything else is mechanical.

## Phase 1 ÔÇö verify cluster register bases against the ADT (the crux)

Before writing any chip-id case, confirm the T6031 cluster bases are right for
T6040. All reads ÔÇö safe.

1. Dump the relevant ADT nodes over the proxy: `/arm-io/pmgr`, the `/cpus`
   cluster nodes, and any `cpu-cluster`/`pmgr`-adjacent nodes carrying cluster
   MMIO regs. Compare the arm-io base + cluster reg offsets to the hardcoded
   `0x21{0,1,2}e00000`.
2. Cross-check by **reading** `CLUSTER_PSTATE` (`base + 0x20020`) at each candidate
   cluster base via the proxy (`p.read64`). A plausible value (BUSY/SET bits sane,
   `pstate_reg_to_pstate()` in-range 1..N) at all three bases = bases confirmed.
   Garbage / bus error = bases wrong ÔåÆ derive the correct ones from the ADT and
   define a new `t6040_clusters[]` instead of reusing `t6031_clusters`.
3. Record the confirmed bases in this file before touching code.

**Exit:** we know, from the machine, the three cluster PSTATE bases for T6040.

### Ô£à Phase 1 RESULT (2026-07-10) ÔÇö bases confirmed, reuse t6031_clusters

Verified live on the M4 (running our chainloaded build). Three ways:
- **Register probe:** guarded reads of `0x210e00000 / 0x211e00000 / 0x212e00000
  + 0x20020` all responded (no fault), value `0x400101` ÔåÆ `DESIRED1=1` (valid
  pstate), decode consistent with the t6031 CLUSTER_PSTATE bit layout.
- **ADT `/cpus`:** cluster windows match the hardcoded bases exactly ÔÇö
  cluster-id 0 (type E) cpus at `0x210_0X0000`, cluster-id 1 (P) at `0x211_0X0000`,
  cluster-id 2 (P) at `0x212_0X0000`. PSTATE reg = cluster window `0x21N000000`
  + `0xe00000`. Matches the measured MPIDR topology (E=cl0, P0=cl1, P1=cl2).
- **Count/types:** 1E + 2P, exactly what `t6031_clusters` (ECPU0/PCPU0/PCPU1)
  expects.

**Confirmed cluster PSTATE bases (== t6031):**
| cluster | type | base |
|---|---|---|
| ECPU0 | E | `0x210e00000` |
| PCPU0 | P | `0x211e00000` |
| PCPU1 | P | `0x212e00000` |

ÔåÆ **Phase 2 simplifies:** no new `t6040_clusters[]` needed; add `case T6040:` to
`return t6031_clusters`. The earlier "bases likely differ" risk is retired.

## Phase 2 ÔÇö code changes

T6040 tracks T6031/T6030 in every chip_id switch. Add a `case T6040:` alongside
`T6031` in all five sites (exact locations as of this commit):

| # | Fn / line | What T6040 joins |
|---|---|---|
| 1 | `pstate_reg_to_pstate` ~L70 | `FIELD_GET(CLUSTER_PSTATE_DESIRED1, val)` group |
| 2 | `set_pstate` ~L110 | `CLUSTER_PSTATE_SET \| DESIRED1` group |
| 3 | `cpufreq_init_cluster` ~L207 | the `write64(base+0x440f8, 1)` group (NOT the T6022 APSC-snooze BIT(40) path) |
| 4 | `cpufreq_get_clusters` ~L412 | `return t6031_clusters` ÔÇö *or* `return t6040_clusters` if Phase 1 found different bases |
| 5 | `cpufreq_get_features` ~L527 | `return t6030_features` |

If Phase 1 says bases differ, add:
```c
static const struct cluster_t t6040_clusters[] = {
    {"ECPU0", <base0>, false, 1, 5},
    {"PCPU0", <base1>, true,  1, 6},
    {"PCPU1", <base2>, true,  1, 6},
    {},
};
```
(Field order: name, base, is_pcore, apsc_pstate, default_pstate ÔÇö copied from the
T6031 entry; adjust only if ADT/pstate data says otherwise.)

## Phase 3 ÔÇö validate on hardware (ÔÜá pmgr writes ÔÇö GATED)

cpufreq init **writes** cluster PSTATE registers. Per session safety rules, any
pmgr write outside the normal SMP path needs sign-off: **before running the write
path I will show the exact address+value list (each `cluster->base + 0x20020` /
`+0x440f8` and the value) and wait for approval.** These are the standard cpufreq
regs, not SPMI/PMU/charger ÔÇö but they are still pmgr writes, so they wait.

1. Read-only dry run first: with the new code built, add temporary prints (or use
   the proxy) to dump what `cpufreq_init` *would* write, without writing. Review.
2. On approval, chainload the new m1n1 and let `cpufreq_init` run at boot.
3. Confirm: boot log prints `cpufreq: Initializing clusters` and **no**
   `Chip 0x6040 is unsupported`; all three clusters init without the
   `CLUSTER_SWITCH_TIMEOUT` warning; `set_pstate` round-trips (read back the
   pstate you set).

### ÔÜá Phase 3 FINDING (2026-07-10) ÔÇö amx-thrtl unsafe on T6040, feature table needs a fork

During the before-snapshot (safe reads), reading `amx-thrtl` (cluster base
`+0x40250`, in `t6030_features`) misbehaved on T6040:
- ECPU0 `0x210e40250`: synchronous fault (guard-skipped).
- PCPU0 `0x211e40250`: **SError** ÔåÆ wedged the proxy; M4 required a power-cycle.

`cpufreq_init` does `mask64` (read-modify-write) on every feature reg per cluster,
so chainloading the current build (which reuses `t6030_features` for T6040) would
very likely SError during cpufreq init at boot. Likely cause: M4 replaced AMX with
SME, so the AMX throttle register at `0x40250` doesn't exist on T6040.

**Revised Phase 2/3 plan:**
1. Add a dedicated `t6040_features[]` = `t6030_features` **minus `amx-thrtl`**
   (and re-examine `llc-thrtl` `0x40270` and `ppt-thrtl` `0x48400/0x48408` on the
   P-clusters ÔÇö the SError aborted the snapshot before those P-cluster reads).
   Point `cpufreq_get_features()`'s `case T6040:` at `t6040_features`, NOT
   `t6030_features`.
2. Investigate whether `pmgr_get_feature("amx-thrtl")` even resolves on T6040 ADT;
   confirm which throttle regs are valid before writing any.
3. Snapshot again (skipping `0x40250`), then re-gate the chainload.

Note: guarded proxy reads catch synchronous aborts but NOT asynchronous SError ÔÇö
avoid raw reads of registers suspected absent; prefer ADT/feature-flag checks.

### ÔÜá Phase 3 FINDING #2 (2026-07-10) ÔÇö whole 0x40xxx throttle region differs on P-clusters

Confirm-read (skipping amx `0x40250`) SError'd again, this time on PCPU0
`llc-thrtl` `0x211e40270` ÔÇö a register that read cleanly (=0) on the E-cluster.
So the divergence is not just amx-thrtl: the t6030 `0x40xxx` throttle offsets
(`ppt/llc/amx`) do NOT map to the T6040 **P-cluster** register layout at all.
E-cluster and P-cluster register maps differ on T6040. Second power-cycle needed.

**Only `CLUSTER_PSTATE` (`+0x20020`) is validated safe on ALL clusters (E+P).**

**Revised approach ÔÇö minimal cpufreq touching ONLY 0x20020:**
- `t6040_features[] = { cpu-apsc (0x20020), cpu-fixed-freq-pll-relock (0x20020) }`
  ÔÇö drop ppt/llc/amx-thrtl entirely (their T6040 offsets are unknown ÔåÆ RE later).
- `cpufreq_get_features()` `case T6040:` ÔåÆ `t6040_features`.
- `cpufreq_get_clusters()` `case T6040:` ÔåÆ `t6031_clusters` (validated).
- pstate switches (`pstate_reg_to_pstate`, `set_pstate`): T6040 in DESIRED1 group
  (0x20020) ÔÇö already done.
- **Line ~214 switch:** do NOT reuse the t6030 case (it does `write64(base+0x440f8,
  1)`, and 0x440f8 is unverified/suspect on P-clusters). Give T6040 its own case
  that just `break`s (no 0x440f8 write). Result: cpufreq_init touches ONLY 0x20020,
  fully validated on every cluster.
- This clears the "unsupported" message and manages P-states; throttles deferred
  to an upstream-coordinated RE pass for the T6040 register map.

### Ô£à Phase 3/4 RESULT (2026-07-10) ÔÇö minimal cpufreq WORKS on T6040

Chainloaded the patched build; verified two paths:
- **Boot path** (`cpufreq_fixup` @ main.c:182): no more `unsupported`; `fixup_cluster`
  correctly no-ops for T6040 (hits `default: return`, only a safe 0x20020 read).
- **`cpufreq_init` path** (exercised via proxy op `p.cpufreq_init()`, since plain
  boot only runs fixup): prints `Initializing clusters`, **retval 0**, no SError.
  `CLUSTER_PSTATE` (only 0x20020 touched) moved `DESIRED1 1ÔåÆ5` (E), `1ÔåÆ6` (P) ÔÇö
  i.e. `cpu-apsc` enabled Apple hardware APSC DVFS and clusters settled at their
  nominal pstates. `BUSY=0`, no switch-timeout ÔåÆ hardware accepted the writes.

Net: cpufreq is functional on T6040 for pstate/APSC management, touching only the
validated `0x20020`. Throttles (ppt/llc/amx) deferred ÔÇö need T6040 register-map RE.

## Risks

| Risk | Mitigation |
|---|---|
| Cluster bases differ from T6031 (likely, given UART base differs) | Phase 1 verifies against ADT + live reads before any write; define `t6040_clusters[]` if so |
| pstate/feature tables subtly differ from T6031 | Read-back validation in Phase 3; watch for CLUSTER_SWITCH_TIMEOUT |
| pmgr write traps or wedges a cluster | Read-only dry run first; writes gated on approval; power-cycle recovers |
| Rust toolchain still broken ÔåÆ can't build stage-2 | Phase 0 checks whether stage-1 bin (chainload target) needs Rust at all; fix rustup only if required |
