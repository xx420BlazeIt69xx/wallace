# T6040 PMGR deterministic matrix (2026-07-12)

This session replaced the blind N=1 PMGR results from session 2 with
reproducible DockChannel-console boots. Every image used the same build #11
kernel and `initramfs-dcuart.cpio.gz`; only the DTB changed. A result is `BOOT`
only when the DebugUSB console printed the Linux banner and reached the BusyBox
prompt. `FAIL` means m1n1 completed the kernel handoff but Linux emitted no
DockChannel banner and the KIS link dropped before userspace.

The variants are generated by `scripts/t6040-pmgr-variant.py` and
`scripts/t6040-build-pmgr-variants.sh`. The latter regenerates the topology from
the committed J614s ADT instead of relying on the already policy-modified kernel
DT source.

PMGR0+1+2 and PMGR0+1+3 deletion variants also failed, including policy
variants. They are **not valid evidence that both controllers independently
fail**: the generated hierarchy contains cross-controller parents, so disabling
a whole controller leaves consumers without providers and reintroduces the
deferred-probe/topology confound. Do not use those results for culprit claims.

## Conclusions

The full 214-domain topology is no longer a boot blocker when the following
temporary raw-boot policy is present:

1. Mark every domain found active at probe time always-on
   (`apple,preserve-active` on all four PMGR controllers).
2. Disable `ps_disp_cpu`.
3. Skip auto-enable writes for `dispext0_{sys,fe,cpu}` and
   `dispext1_{sys,fe,cpu}`.

All five ANE exclusions in the older functional DT are unnecessary. Both
dispext banks are necessary at the current granularity. The next clean split is
within each bank's `sys`/`fe`/`cpu` members.

The old curated PMGR result is also explained: its domain-class removals were
not the problem; reparenting/flattening the surviving PMGR1 hierarchy was.

## Generator fixes

`pmgr_adt2dt.py` previously treated Apple's `critical` flag as Linux
`apple,always-on`. That is not a valid policy mapping: on M4 it marks PMC and
PMS/FPWM domains but misses AIC, opposite the curated t8132 policy. The generator
now emits no always-on policy by default; `--always-on critical` exists only for
legacy experiment reproduction.

It also no longer emits parent phandles to `no_ps` devices. The J614s ADT lists
`dispext_rail_v` as a parent of two managed display domains even though that
virtual parent has no power-state register and is never emitted.

Commits: m1n1 `5dc76503`; curated `t6040-bringup` worktree `effcc16c`.

## Trackpad side finding

`event0` is confirmed as Apple DockChannel Multi-touch and `event1` as the
keyboard, but trackpad reads fail even immediately after a fresh boot. The first
open sends two command `0x40` interface resets; firmware returns `0xe00002c2`
for both, and `dchid_open()` times out. Because `iface->starting` is never
cleared, the next open returns `-EINPROGRESS` immediately. Haptic/tactile
feedback is a separate actuator interface and is not required for motion
events.
