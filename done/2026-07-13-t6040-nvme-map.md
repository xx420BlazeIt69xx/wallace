# T6040 J614s ANS/NVMe map (2026-07-13)

## Result

The complete storage-controller map was read from the live J614s ADT through
the m1n1 proxy without reading or writing any MMIO. T6040 follows the T8140
ANS2 shape at different addresses. It uses a power-managed CoastGuard SART v3
and the NVMe controller's embedded NVMMU; there is no Apple DART in this path.

Linux commit `9cf4a92fa16f` adds the following nodes to `t6040.dtsi`, all
disabled so the standard DTB performs no new hardware access:

The table below records the raw ADT values. The live `/arm-io` range translates
them by `+0x200000000`; Linux therefore uses CPU PAs `0x409600000`,
`0x409608000`, `0x40dc50000`, and `0x40dcc0000`.

| Component | ADT address / size | IRQs | Source |
|---|---:|---:|---|
| ANS ASC control | `0x209600000 / 0x4000` | — | ADT `/arm-io/ans` reg 0 |
| ANS mailbox | `0x209608000 / 0x4000` | `1530..1533` | ASC + `0x8000`; ADT raw order `[1531,1530,1533,1532]` |
| SART v3 | `0x20dc50000 / 0xc000` | — | ADT `/arm-io/sart-ans` reg 0, `sart-version = 3` |
| SART power control | `0x20dcc13e8 / 0x4` | — | ADT reg 2 `0x20dcc0000` + `sart-power-reg-offset = 0x13e8` |
| NVMe + NVMMU | `0x20dcc0000 / 0x60000` | `2583` | ADT `/arm-io/ans` reg 3 and `nvme-interrupt-idx = 4` |

The ANS node's role is `ANS2`; its first four raw IRQs use Apple's not-empty
first ordering and were normalized to the Linux mailbox order. The power path
is `ps_ans` plus `ps_apcie_phy_sw`; the latter already depends on both storage
PCIe system domains. The provisional compatibles follow yuka's existing T8140
WIP nodes. ANS2 follows T8103, but the probe proved that T8140 SART is not
safely compatible with the unmodified T6000 driver despite sharing its v3
entry layout: the register file must first be activated separately.

The current standard `t6040-j614s-dcuart.dtb` keeps every storage node disabled,
so the extra resource description performs no hardware access. Its SHA-256 is
`bc1534d214d6cde172b1aad1b78f85104f9c0a9790a6fabd1e9990fe950ff41b`;
it was boot-verified back to the build #15 BusyBox shell after the probe.

## Approved first-probe result

The approved built-in image plus full DTB handed off successfully but never
reached the Linux DockChannel shell. KIS disconnected after the watchdog-reset
interval. The unchanged standard image and DTB immediately booted to BusyBox,
excluding a tether, PMGR, or general kernel regression.

A staged rebuild put `nvme-core` and `nvme-apple` in the initramfs as modules.
The same full DT still reset before userspace without either module loaded, so
the failure preceded Apple NVMe probe. Two cumulative isolation DTs then made
the boundary exact:

- ASC mailbox only reached BusyBox and reported the gated mailbox model;
- ASC mailbox plus SART, with NVMe disabled, reset before userspace.

Therefore the first probe stopped at SART. The namespace never enumerated. No
`fdisk`, partition, mount, filesystem, flush, or other disk command was issued,
and no ANS/NVMe register write was reached. The target was returned to the
standard build #15 BusyBox image after the investigation.

## CoastGuard power-control finding

A second read-only m1n1 ADT dump, with no MMIO access, showed that
`/arm-io/sart-ans` has all of the following:

- `sart-version = 3`, `compatible = "sart", "coastguard"`;
- the boolean `sart-power-managed`;
- `sart-power-reg-offset = 0x13e8`;
- reg 0 `0x20dc50000 / 0xc000` for entries; and
- reg 2 `0x20dcc0000 / 0x4000`, placing the control register at
  `0x20dcc13e8`.

Static disassembly of this machine's paired AppleSART driver from the macOS
26.5.2 (25F84) kernelcache established the exact lifecycle without touching
the target. `IOCoastGuardSARTMapper` maps provider memory index 2. On its first
active mapping it repeatedly writes `0`, delays 100 us, and reads back until
the register is `0`. On the last unmap it repeatedly writes `1`, delays 100 us,
and reads back until it is `1`. Both paths are protected by a lock and active
mapping reference count. No Apple binary or extracted kernelcache is stored in
this repository.

This corrects the initial T6000 fallback audit: the v3 entry format is shared,
but reading the entry register file while CoastGuard reports inactive triggers
the observed reset.

## Remaining ANS/NVMe audit

The other fallback choices were compared line-by-line with both Linux and
m1n1:

- `apple,asc-mailbox-v4` selects the standard ASC FIFO layout used by the live
  ADT's four mailbox interrupts;
- the T8140 SART uses the v3 address/size/flags layout, now combined with its
  separately derived CoastGuard lifecycle instead of a T6000 fallback;
- `apple,t8103-nvme-ans2` selects 64-entry linear submission queues and the
  embedded NVMMU; and
- Linux's boot-status, queue-doorbell, controller, and NVMMU programming
  offsets match m1n1's implementation.

There is one old diagnostic-only source discrepancy: m1n1 defines the TCB
status read as `0x29120`, while Linux has used `0x28120` since its original
2022 Apple NVMe driver. m1n1 already used its value on pre-T6040 machines, so
this is not evidence of a T6040 register move. The read only warns after a TCB
invalidation; it does not select or trigger invalidation and does not change
the first-probe plan. Do not resolve it by probing either offset live without a
separately reviewed register source.

Proceeding beyond the SART-only retry invokes these bounded driver writes:

- PMGR transitions for `ps_ans` and `ps_apcie_phy_sw`;
- CoastGuard state requests at exactly `0x20dcc13e8`: `0` for active and `1`
  for inactive, with 100 us delay and matching readback;
- SART v3 entry programming below `0x20dc500c0`, with physical address and size
  values chosen from dynamically allocated RTKit buffers;
- ASC RUN at `0x209600044` and mailbox traffic in `0x209608000..0x209608fff`;
- NVMe/NVMMU setup registers below the ADT-backed
  `0x20dcc0000..0x20dd1ffff` resource, including dynamically allocated queue and
  TCB addresses; and
- ordinary NVMe identify/admin commands after firmware boot.
