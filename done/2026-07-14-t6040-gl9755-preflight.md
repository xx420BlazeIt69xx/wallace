## Endpoint and DT audit

The paired J614s ADT inventory in
`done/2026-07-14-t6040-wireless-pcie-map.md` identifies the port-1 endpoint as
Genesys Logic `17a0:9755`, with PERST# GPIO 5, CLKREQ# GPIO 1 function 2, an
ADT bridge cap of Gen1, and its own T8110 DART at `0x411000000`, IRQ 1733.

`dts/t6040-j614s-dcuart-pcie.dts` carries that topology without adding an
address or register:

- root port `pci@1,0`, config-space BDF encoding `reg = <0x800 ...>`;
- `reset-gpios = <&pinctrl_ap 5 GPIO_ACTIVE_LOW>`;
- `max-link-speed = <1>` and `bus-range = <2 2>`;
- endpoint `mmc@0,0`, `compatible = "pci17a0,9755"`, BDF encoding
  `reg = <0x20000 ...>` (bus 2, device 0, function 0);
- `cd-inverted` and `wp-inverted`, matching the existing Apple GL9755 nodes in
  `t600x-j314-j316.dtsi` and `t600x-j375.dtsi`;
- `iommu-map` entry `<0x200 &pcie0_dart_1 1 1>` under mask `0xff00`, mapping
  bus-2 requester IDs to port-1 DART SID 1;
- DART base `0x411000000`, size `0x20000`, IRQ 1733.

The already prepared PCIe DTB, SHA-256
`258ca9e52284864498dde10cabc807f94990528cfc7e3dad98356bf999ff2eb2`, was
decompiled again. It retains all of the properties above, including GPIO 5,
Gen1, bus 2, `pci17a0,9755`, both inversion flags, and the `0x200` DART map.
`dtc` emitted only the pre-existing `/soc/serial` simple-bus warning, unrelated
to PCIe or GL9755.

## Mainline driver coverage

The local mainline tree already has explicit GL9755 support:

- `sdhci-pci-core.c` matches `PCI_VENDOR_ID_GLI` plus
  `PCI_DEVICE_ID_GLI_9755` and selects `sdhci_gl9755`;
- the fixup sets `SDHCI_QUIRK_NO_ENDATTR_IN_NOPDESC` and
  `SDHCI_QUIRK2_BROKEN_DDR50`;
- the GL9755 probe enables MSI, SDHCI v4 mode, and the UHS-II-aware host path,
  and marks SDIO unsupported;
- its Apple-specific code consumes the DT `cd-inverted` and `wp-inverted`
  flags and sets the corresponding GL9755 vendor bits.

With the gated PCIe configuration, `CONFIG_MMC`, `CONFIG_MMC_BLOCK`, and
`CONFIG_MMC_SDHCI` are built in; `CONFIG_MMC_SDHCI_PCI=m` selects
`CONFIG_MMC_SDHCI_UHS2=m`. A W=1 compile of `sdhci-pci-core.o` and
`sdhci-pci-gli.o` passed. The current image recipe intentionally does not run a
full modules build or stage either module in its initramfs, so no loadable
module artifact is claimed here. A later card-I/O stage must build the normal
module set together (including `sdhci-uhs2.ko`), record hashes, and stage both;
building only `sdhci-pci.ko` cannot satisfy modpost without its selected
UHS-II dependency and the kernel's module symbol inventory.

That separation is useful: GL9755 probe performs ordinary endpoint PCI config,
MSI, clock, power, and vendor-register writes. It must remain unbound during
the first link-detection run.

## Read-only userspace discriminator

`scripts/t6040-gl9755-readonly-check.sh`, SHA-256
`ea122f245a7faf585b808d303331367f772f434c5ecd7e973e9bf022fae3a0ed`, is a
`/bin/sh` test suitable for the minimal initramfs. It uses only shell tests,
`cat`, and `ls`. It requires:

- PCI function `0000:02:00.0` to exist;
- vendor/device `17a0:9755` and SD-host class `0x0805xx`;
- no bound endpoint driver;
- an IOMMU-group link, exercising the expected DART attachment;
- optional reporting of negotiated link speed and width.

The script never writes sysfs, opens an MMC block device, loads a module, or
touches an SD card. It should be staged as `/t6040-gl9755-readonly-check.sh`
only in a future exact, hashed initramfs, then invoked after the DockChannel
shell appears:

```sh
/bin/sh /t6040-gl9755-readonly-check.sh
dmesg
```

the test adds no GL9755 vendor-register or media access.
