## Key difference from cpufreq/MCC: NOT boot-exercised, NOT locally testable

`pcie_init()` is called **only at kboot** (`src/kboot.c:2869`, Linux-boot handoff),
NOT at m1n1 boot (`main.c` only includes the header). So:
- There is **no boot-log "unsupported" gap** — a chainload won't run this code.
- Validation is ADT-derivation only for now; the real bring-up is **testable
  incrementally against Stage C** (per the roadmap), when we boot a kernel.
- **Do NOT run `pcie_init` live on the M4.** It does invasive MMIO/tunable/power
  writes and would be a far bigger poke than the single MCC read that wedged us.
  Zero upside before Stage C. (Advisor-confirmed.)

## Live ADT findings (`/arm-io/apcie0`, hardware-verified 2026-07-10)

- Node is **`/arm-io/apcie0`** (not `/arm-io/apcie`), compatible **`apcie,t6040`**.
  `#ports=4`, `lane-cfg=0`, `#msi-vectors=32`, 2 populated bridges (pci-bridge0/1).
- **35 reg entries.** Shared block = **reg[0..6]** (see below); the remaining 28
  are **4 ports × 7 regs each** — a clean, identical per-port pattern
  (0x410xxxxx / 0x411 / 0x412 / 0x413 blocks + 2 per-port phy windows in 0x417).
- Shared reg index mapping (translated via `node.get_reg(i)`), matches
  `regs_t6031`'s indices exactly:

  | idx | base | size | role (regs_t6031) |
  |---|---|---|---|
  | 0 | 0x1cb0000000 | 0x10000000 | config / ECAM |
  | 1 | 0x414000000 | 0x4000 | rc |
  | 2 | 0x417000000 | 0x40000 | phy_common / phy |
  | 3 | 0x417040000 | 0x28000 | phy_ip |
  | 4 | 0x416000000 | 0x1000000 | axi |
  | 5 | 0x415046200 | 0x4000 | **extra — not used by base init** |
  | 6 | 0x415044000 | 0x4000 | **extra — not used by base init** |

- **THE reuse-breaker: `shared_reg_count`.** `regs_t6031` uses 8; with 8,
  `port_regs = 35 - 8 = 27`, and `27 % 4 != 0` → the even-divide check
  (pcie.c:386) **fails**. t6040 has **7** shared regs (28/4 = 7 per port, verified
  by the repeating port pattern). ⇒ needs `regs_t6040` = `regs_t6031` with
  `shared_reg_count = 7`. Everything else (indices 0..4, T8122-family init path)
  is identical.
- **num_phys = 1** (single phy reg window; `apcie-phy-tunables` targets only
  reg[2], and the pll/auspma tunables use the no-`%d` names → the `num_phys==1`
  code path). Do NOT route `lane-cfg=0` through the GE branch's lane→num_phys
  logic — that's for `apcie-ge` only.

## Tunables (decoded from ADT; format = {u32 off; u32 size; u64 mask; u64 val})

Applied by existing code (all present on t6040, so reuse works):
- `apcie-axi2af-tunables` (77 entries, → axi_idx) ✓
- `apcie-phy-tunables` (5 entries, → phy_idx) ✓
- `apcie-phy-ip-pll-tunables` (28, → phy_ip_base) ✓
- `apcie-phy-ip-auspma-tunables` (204, → phy_ip_base) ✓
- `apcie-common-tunables` — **absent on t6040** (code skips it cleanly; fine).

**NEW, unhandled by any current code — the real remaining RE:**
- `apcie-cio3pllcore-tunables` (7 entries, offsets 0x0..0x100).
- `apcie-pcieclkgen-tunables` (1 entry, off 0x0, mask 0x3e0 val 0x220).

Their offsets are tiny (fit any window), so the **target reg window is NOT
derivable from the ADT alone**. Strong candidates: **reg[5] (0x415046200) and
reg[6] (0x415044000)** — the two extra shared windows the base init never touches,
and the names are clock/PLL-ish (`cio3pllcore`, `pcieclkgen`). But which tunable →
which window is a guess, and a wrong target = async SError at Stage C. Per the
safety rule + advisor: **leave documented-but-stubbed (unapplied) — do NOT guess a
target window.** Not applying them is safe; the base init already skips unknown
tunables. Resolve at Stage C (gated bring-up) or via a booted-macOS reg diff.

## Phase 1 — add the t6040 branch (ADT-derived, no live run) ✅ scoped here

1. Add `regs_t6040` (= `regs_t6031`, `shared_reg_count = 7`) and dispatch
   `apcie,t6040` → it. type=APCIE_T6031 / compat=APCIE_T8122 so every existing
   `type==APCIE_T6031` / `compat==APCIE_T8122` code path fires identically to the
   validated t6031 template (minimises behavioural divergence — we can't test
   locally, so stay as close to known-good as possible).
2. Do NOT reference the two new tunables (they stay skipped). Document them.
3. Cannot chainload-test (kboot-only). Build must be clean; correctness is by
   ADT-derivation + faithful t6031 reuse.

**Exit:** code compiles; `regs_t6040` matches the verified ADT reg layout; branch
present; new tunables documented as the open RE item.

## Phase 2 — Stage C bring-up (deferred, gated)

When kboot runs on a real kernel boot:
1. Gate the first `pcie_init()` (show writes / step through) per the pmgr-write
   safety rule. Watch for link-up on ports 0/1 (the populated bridges).
2. If link training fails, the missing `cio3pllcore` / `pcieclkgen` tunables are
   the prime suspect — resolve their target window (try reg[5]/reg[6], gated,
   one at a time; async-SError ready) and wire them in as a `regs_t6040`-only
   step.
3. `dt_set_pcie_tunables()` (kboot.c:1324) — check it copies the right t6040
   tunables into the FDT for the kernel's apple-pcie driver.

## Risks

| Risk | Mitigation |
|---|---|
| Running pcie_init live wedges/SErrors the daily-driver | Don't. kboot-only; defer to Stage C, gated |
| Wrong shared_reg_count → port-reg misparse / wrong MMIO | Verified 7 from the repeating 4×7 port pattern; even-divide check guards it |
| New tunables unhandled → link won't train at Stage C | Documented; safe to skip now; resolve gated at C. Not applying ≠ SError; guessing a window = SError |
| t6040 phy/clk genuinely differs from t6031 beyond regs | Only discoverable at Stage C; stay faithful to t6031 until then |
