# Next step — T6040 MCC bring-up (Stage B, item 2)

Written 2026-07-10, after cpufreq (item 1) landed. Detailed plan for the roadmap's
Stage B item 2. Companion: `roadmap.md`, `2026-07-10-t6040-cpufreq-plan.md` (the
template + hard-won SError lessons).

## Why this is next

Second (and last) gap the boot log names: `MCC: Unsupported version:mcc,t6041`.
MCC = the memory cache controller (System Level Cache + TrustZone carveout +
DCS/DRAM channel config). m1n1 needs it for the framebuffer/coprocessor carveouts
it hands to Linux, and SLC enable matters for memory bandwidth. Same low-RE,
reuse-the-M3-template shape as cpufreq.

## Current state (src/mcc.c, verified 2026-07-10)

`mcc_init()` (src/mcc.c:388) dispatches on the ADT `/arm-io/mcc` compatible string:
t8103 / t8112 / t6000 / t6020 / t6031 → their init fns; **anything else →
"Unsupported"**. T6040's node is `mcc,t6041`, which has no branch.

`mcc_init_t6031()` is **ADT-driven and touches NO MMIO at init** — it only reads
`reg`, `dcs-count-per-amcc`, `plane-count-per-amcc` and stores bases/strides into
`mcc_regs[]` using the `T6031_*` offset constants. So adding a t6041 branch and
running it at boot is **low-risk** (no hardware poke — unlike cpufreq's throttle
regs). The MMIO writes happen later and elsewhere:
- `mcc_enable_cache()` — kboot.c:2864 (Linux-boot handoff only) → writes
  `PLANE_CACHE_ENABLE` (0x1c00) per plane. **Not exercised until we boot a kernel.**
- `mcc_unmap_carveouts()` — memory.c:451.

T6031 constants to validate for t6041: `PLANE_OFFSET=0`, `PLANE_STRIDE=0x40000`,
`GLOBAL_OFFSET=0x100000`, `DCS_OFFSET=0x400000`, `DCS_STRIDE=0x200000`,
`CACHE_WAYS=12`, `CACHE_STATUS_MASK/VAL`, and `t603x_tz_regs`
(count=4, stride=0x14, start=0x6d8, end=0x6dc, enable=0x6e4).

## Phase 1 — add the t6041 branch (ADT-only, low-risk)

1. In `mcc_init()`, add:
   ```c
   } else if (adt_is_compatible(adt, node, "mcc,t6041")) {
       return mcc_init_t6031(node, path);
   ```
   (Reuse t6031 init; t6041 is the M4 Pro analogue of t6031/M3 Max, same as the
   cpufreq cluster reuse.)
2. Rebuild, chainload. `mcc_init` runs at boot (ADT-only). **Expected:**
   `MCC: Initialized T6031 MCCs (N instances, P planes, C channels)` and no
   "Unsupported". No writes, so no SError risk at this step.
3. Sanity-check the reported instance/plane/channel counts against the ADT
   (`/arm-io/mcc` `dcs-count-per-amcc`, `plane-count-per-amcc`, and reg count).
   If mcc_init_t6031 mis-parses (wrong counts / adt_get_reg failure), that's the
   signal the t6041 ADT layout differs → may need a `mcc_init_t6041` variant.

**Exit:** boot log shows MCC initialized with plausible counts; no "unsupported".

### Phase 1 — DONE (2026-07-10)

The naive "t6041 → mcc_init_t6031" reuse does **not** work: it mis-parses the
t6041 ADT. Verified on the live M4 Pro. Findings:

- **t6041's `/arm-io/mcc` reg map differs from t6031.** t6031 hardcodes
  `reg_offset=3` and takes `reg[3..end]` as the AMCC windows → on t6041 that gave
  **13 bogus instances** spanning the SOCNI/PMP windows. t6041 states the AMCC
  location/count explicitly in the ADT:
  - `amcc-reg-idx = 12`, `amcc-count = 4` (`amcc-enable-mask = 0xf`).
  - The real AMCC windows are `reg[12..15]` = `0x20000000 + i*0x2000000`, each
    **32 MB** (`size 0x2000000`).
  - `reg[0]`=panic-save, `reg[1]`=mcw, `reg[2..9]`=8× SOCNI (`socni-reg-idx=2`,
    `socni-count=8`, 32 KB each), `reg[10]`=pmp_ptd_db, `reg[11]`=pmp_ptd_update.
  - `dcs-count-per-amcc = 4`, `dcs-channel-enable-mask = 0xffff` (4 AMCC × 4 = 16).
- **No `plane-count-per-amcc` property** on t6041 (t6031 has it). The t6031 code's
  error check is inverted (`if (!ADT_GETPROP(...))` — but `adt_getprop_copy`
  returns a *negative* error, so `!(-N)==0` swallows the failure), so the missing
  prop silently produced `plane_count=0` ("0 planes") instead of an error.
- **`amcc-count` is a 64-bit property** (`len=8`), unlike `amcc-reg-idx` /
  `dcs-count-per-amcc` (`len=4`). Must be read into a `u64` or `ADT_GETPROP`
  returns `BadLength`. (Cost me one chainload — "Failed to get amcc-count!".)

**Fix landed:** new `mcc_init_t6041()` (src/mcc.c) — ADT-driven, derives
`reg_offset` from `amcc-reg-idx`, instance count from `amcc-count`, channels from
`dcs-count-per-amcc`; reuses `T6031_*` offsets + `t603x_tz_regs`; uses correct
(`< 0`) error checks; flags the missing plane count explicitly and leaves
`plane_count=0` (cache-enable is inert at boot anyway). Boot now prints:
`MCC: Initialized T6041 MCCs (4 instances, 0 planes, 4 channels)`, no "unsupported",
no SError, proxy alive.

**ADT-side offset sanity (done, no MMIO):** all four AMCC windows are 32 MB;
every T6031-derived offset fits — plane 0, global `0x100000`, dcs up to
`0xa00000` (=`0x400000 + 3*0x200000`), cache-enable `0x1c00`. So the derived
addresses land inside mapped windows (Phase 2 item 1 pre-req satisfied).

**Open for Phase 2:** the per-AMCC **plane count** (ADT gives none) — must be
resolved before `mcc_enable_cache()` (Stage C) via careful SError-aware probing
of `PLANE_CACHE_STATUS` (0x1c04). Until then `plane_count=0` = no planes enabled.

## Phase 2 — validate the register offsets (careful, SError-aware)

Before trusting `mcc_enable_cache()` (which WILL write MMIO at kboot), verify the
T6031 offsets map to t6041 — but heed the cpufreq lesson: **raw MMIO reads of
wrong offsets can raise async SError that the proxy guard does NOT catch, wedging
the link (power-cycle).**
1. Prefer ADT/derivation over blind probing: confirm the `reg` ranges and derived
   plane/global/dcs bases are inside mapped MCC windows first.
   **[DONE in Phase 1]** — 4× 32 MB AMCC windows at `reg[12..15]`; all T6031
   offsets fit. The remaining unknown is the **plane count** (see Phase 1 notes):
   probe `PLANE_CACHE_STATUS` (base+0x1c04, then +plane_stride 0x40000 per plane)
   with a guarded read per candidate plane index until it stops reading back a
   valid status, to count real planes. One guarded read at a time, power-cycle
   ready.
2. If probing MCC registers directly, do ONE careful guarded read of a benign
   register (e.g. `PLANE_CACHE_STATUS` 0x1c04) per instance, ready to power-cycle.
   Do NOT sweep unknown offsets.
3. `mcc_enable_cache()` is only called at kboot (Linux boot), which we are not
   doing yet — so its writes are **out of scope until Stage C**. Gate them (show
   address+value, wait) when we get there, per the pmgr-write safety rule.

**Exit:** confidence that t6041 SLC/carveout layout == t6031, OR a documented list
of offset deltas needing a t6041 variant.

### Phase 2 — DONE for SLC/plane (2026-07-10); TZ offset deferred to Stage C

**Outcome:** the two register groups that matter for `mcc_enable_cache()` (Stage C)
are resolved and encoded in `mcc_init_t6041()`; the TZ/carveout offset is a
documented open delta (not needed until Stage C, m1n1 boots fine without it).

**Code landed (src/mcc.c):** `T6041_CACHE_STATUS_MASK/VAL = 0x00010101`,
`T6041_PLANE_COUNT = 1`; `mcc_init_t6041` now sets `plane_count=1` and the t6041
status mask/val. Boots clean: `Initialized T6041 MCCs (4 instances, 1 planes, 4
channels)`, proxy alive.

**Plane count = 1 (was the big open item):** the four AMCCs each expose a single
cache plane. Evidence: `PLANE_CACHE_ENABLE` (0x1c00) reads 1 on all 4 AMCCs, but
the T6031 plane-1 offset (`base+0x40000+0x1c00`) is unbacked and raised an async
SError (wedged the proxy). So the AMCC register block is compact (< 0x40000) and
the T6031 "4 planes × 0x40000" model does not apply; parallelism is the 4 AMCCs.

**Cache status format = t6041-specific:** enabled status is `0x00010101` (bits
0/8/16), NOT the T6031 12-way DATA(GENMASK 13:9)/TAG(GENMASK 8:4) encoding — that
decodes to 0/16 here, so the T6031 mask/val would hang the enable-poll at Stage C.
Matching `0x00010101` directly. Way count unknown (not needed for the poll).

**TZ / carveout — OPEN (Stage C):** `t603x_tz_regs` offsets (0x6d8/0x6dc/0x6e4 ×4)
are *readable* at AMCC[0] plane 0 but read all-zero. Yet the iBoot ADT
`/chosen/carveout-memory-map` has the TZ carveouts the mcc.c:188 comment points at:
`region-id-2` = base 0x105d9ce4000 size 0x25800000, `region-id-4` = base
0x105ce7a8000 (= "Top of normal RAM") size 0x1f30000. If the TZ offsets were
correct, the start reg would read the RAM-relative page (e.g. `(0x105d9ce4000 -
dram-base 0x10000000000) >> 12 = 0x5d9ce4`). Reading 0 ⇒ **either the TZ offsets
differ on t6041, or M4 conveys these carveouts elsewhere.** m1n1 boots fine (it
just doesn't unmap them), so this is a Stage-C correctness item. **Resolve via**
a *targeted, bounded* value-search of the AMCC low register block for 0x5d9ce4 /
0x5ce7a8 (stay < 0x40000 where reads are backed; one guarded read at a time,
power-cycle ready), OR a booted-macOS AMCC reg dump. Do NOT blind-sweep.

### (superseded) Phase 2 — partial, then wedged the proxy

Probed AMCC[0] (translated CPU base **0x220000000** = ADT child 0x20000000 +
`/arm-io` ranges 0x200000000; note proxyclient `node.reg[i].addr` is UN-translated
— use `node.get_reg(i)`). Findings, all guarded 32-bit reads:

- **TZ regs (0x6d8/0x6dc/0x6e4, ×4 @ stride 0x14) read cleanly, all `0x0`.** So
  the t603x TZ offsets are *mapped/readable* on t6041 but hold no enabled carveout
  at the m1n1 stage (matches the clean boot: `mcc_unmap_carveouts()` printed no
  "Unmapping TZ" lines). Inconclusive whether offsets are *correct* or just
  happen to read zero — needs the macOS `/chosen/carveout-memory-map` cross-ref
  (per the mcc.c:188 comment) to confirm.
- **`PLANE_CACHE_ENABLE` (0x1c00) = 0x1** → SLC is already enabled by iBoot, and
  this offset is correct (reads back the enable bit).
- **`PLANE_CACHE_STATUS` (0x1c04) = 0x00010101** → readable, but does NOT match the
  T6031 status pattern. T6031 decode (DATA_COUNT=GENMASK(13,9)=12, TAG_COUNT=
  GENMASK(8,4)=12) gives 0/16 here — garbage. ⇒ **`mcc_enable_cache()` with the
  T6031 `cache_status_mask/val` would TIME OUT at Stage C.** t6041 needs its own
  status decode (0x00010101 looks like one bit set per byte at bits 0/8/16 — TBD).
- **⚠️ PLANE SWEEP WEDGED THE PROXY (async SError).** Reading
  `base + plane*0x40000 + 0x1c00` for plane ≥ 1 raised an async SError (guard
  can't catch it) → UartTimeout → proxy dead → **power-cycle**. Plane 0 read fine
  moments before. ⇒ **The T6031 plane model is WRONG for t6041**: either there is
  only **1 plane per AMCC**, or the plane stride is not 0x40000. Blind striding at
  0x40000 is unsafe. This is the biggest delta and it is a *live hazard*:
  `mcc_enable_cache()`'s `for plane` loop would itself SError on real hardware if
  given plane_count>1 with stride 0x40000. **Keeping plane_count=0 (Phase 1) was
  the right call.**

**Deltas confirmed so far (t6041 ≠ t6031):** (a) status register format at 0x1c04;
(b) plane count/stride. Offset *bases* (enable 0x1c00, TZ 0x6d8+) appear correct.

**Next (after power-cycle), lower-risk approaches — do NOT resume blind striding:**
1. Get the real plane count from a non-probing source: macOS ADT / a booted-macOS
   `/chosen` carveout map, or diff the t6031 ADT's `plane-count-per-amcc` value and
   reason about the t6041 chop ratio. If t6041 AMCC == 1 plane, set plane_count=1.
2. To read PLANE_CACHE_STATUS on planes safely, first CONFIRM the plane stride
   (don't assume 0x40000). If only 1 plane, no sweep needed.
3. Decode 0x00010101 into a t6041 `cache_status_mask/val` (probably just "enabled")
   and add T6041_CACHE_* constants; wire into `mcc_regs[].cache_status_*`.
4. All of this is still Stage-C-gated for the actual `mcc_enable_cache()` writes.

## Risks (cpufreq lessons applied)

| Risk | Mitigation |
|---|---|
| t6041 ADT layout differs from t6031 (counts/reg order) | Phase 1 prints counts; cross-check ADT before trusting |
| T6031 offset constants wrong for t6041 → SError when cache enabled | Phase 2 validates offsets; enable-cache deferred to Stage C + gated |
| Async SError from wrong-offset MMIO reads wedges proxy | Don't blind-probe; derive from ADT; one careful guarded read at a time |
| MCC mis-init corrupts carveouts handed to Linux | Only matters at kboot (Stage C); validate counts now, writes later |
